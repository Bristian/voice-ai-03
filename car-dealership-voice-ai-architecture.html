02 Telephony Layer — Vonage Voice API
// Call Control Object
NCCO — Call Routing Logic
When the sales rep doesn't answer, Vonage executes an NCCO (Nexmo Call Control Object) that connects the caller to the AI agent via WebSocket. 
[
  {
    "action": "talk",
    "text": "Please hold...",
    "bargeIn": false
  },
  {
    "action": "connect",
    "endpoint": [{
      "type": "websocket",
      "uri": "wss://api.yourdomain.com/ws/voice",
      "content-type": "audio/l16;rate=16000",
      "headers": { "call_id": "{{call_id}}" }
    }]
  }
]
// Vonage Webhooks
Event Callbacks & State Management
  • POST /webhooks/answer — Returns NCCO, creates Redis session keyed by call_uuid
  • POST /webhooks/events — Handles ringing, answered, completed, failed states
  • POST /webhooks/fallback — Catches NCCO fetch failures, plays static hold message
  • DTMF input captured via input action as fallback for bad audio
  • Call recording optional — stored to S3 for compliance/QA

Vonage JWT authentication; server-side token signed with application_id + private key. Never expose private key to client.
03 EchoKit Server & WebSocket Audio Pipeline
🔊
EchoKit Server is the real-time audio orchestration core — a Node.js/Python process managing the bi-directional WebSocket with Vonage, routing raw PCM audio chunks to ElevenLabs STT and streaming TTS audio back. It also handles Voice Activity Detection (VAD), barge-in interruption, and session state.
// Audio Ingestion
WebSocket Audio Server
  • Built on ws (Node) or websockets (Python asyncio)
  • Accepts raw audio/l16;rate=16000 binary frames from Vonage
  • Frame buffer: 20ms chunks (320 bytes at 16kHz mono)
  • VAD gate: silero-vad or webrtcvad detects speech boundaries
  • On speech end: flush buffer to STT pipeline
  • Connection keyed to call_uuid from Vonage header
// Voice Activity Detection
VAD + Barge-In
  • Barge-in: When caller speaks while TTS is playing, immediately cancel TTS stream and process new utterance
  • Silero VAD runs locally in process — 5ms latency, no network hop
  • Endpointing: 600ms silence = utterance complete
  • Min speech duration: 200ms to filter noise
  • Send BARGE_IN event to Vonage to halt playback
// Audio Output
TTS → WebSocket Streaming
  • OpenAI TTS streams MP3 chunks over HTTP/2
  • Transcode MP3 → PCM 16kHz with ffmpeg pipeline (streaming, not file-based)
  • Push PCM binary frames back to Vonage WebSocket as fast as decoded
  • First audio chunk latency: ~150–200ms
  • Cache TTS audio blobs in Redis by text hash (TTL 24h)
// EchoKit Server core loop (simplified Node.js)
const handleVonageWS = async (ws, req) => {
  const callId  = req.headers['x-call-id'];
  const session = await createSession(callId);
  const vad     = new SileroVAD({ sampleRate: 16000 });
  const sttStream = elevenlabs.streamingSTT({ language: 'en' });

  ws.on('message', async (pcmChunk) => {
    const isSpeech = await vad.process(pcmChunk);
    if (isSpeech) {
      sttStream.write(pcmChunk);           // push to ElevenLabs
      if (session.ttsPlaying) bargeIn(ws); // interrupt TTS
    }
  });

  sttStream.on('transcript', async (text) => {
    const response = await agentPipeline(text, session);
    await streamTTSToWS(ws, response, session);
  });
};
04 Speech Pipeline — ElevenLabs STT + OpenAI TTS
// Speech-to-Text
ElevenLabs Streaming STT
  • Use ElevenLabs Speech-to-Text WebSocket API for sub-200ms streaming transcription
  • Send 20ms PCM chunks continuously; receive partial + final transcripts
  • Enable word_timestamps for accurate barge-in detection
  • Language: en with auto-detection fallback for multilingual callers
  • Custom vocabulary: inject dealership-specific terms (make/model names, trim levels)
  • Confidence threshold: reject transcripts below 0.75, ask for repeat

Fallback: If ElevenLabs is unavailable → OpenAI Whisper API (batch, ~500ms latency penalty).
// Text-to-Speech
OpenAI TTS Streaming
  • Use tts-1 model with alloy or nova voice (professional, warm tone)
  • Streaming via openai.audio.speech.create(stream=True)
  • Output format: pcm (raw 24kHz) → downsample to 16kHz for Vonage
  • Speed: 0.95 — slightly slower than default for phone clarity
  • Pre-generate and Redis-cache: greetings, hold phrases, disclaimers, common FAQ answers
  • SSML-like pauses: insert natural pauses around prices, model names

Cache hit rate target: 40%+ on common phrases reduces TTS latency dramatically.
05 AI Agent Pipeline — Intent, RAG & Voice-to-SQL
🤖
Two-track architecture: Every utterance is classified as either a semantic/FAQ query (handled by RAG over document embeddings) or a structured vehicle search (handled by the Voice-to-SQL agent querying PostgreSQL). Both tracks run in parallel and the fastest valid result wins, or results are merged.
Intent Classification
LLM call with a lightweight system prompt classifies utterance into: vehicle_search, pricing_query, availability_check, financing_inquiry, dealership_info, appointment_request, human_transfer. Cached with Redis for 5min per session to avoid re-classifying follow-ups. Uses GPT-4o-mini for speed (~80ms).
Entity Extraction
Extract structured entities from transcript: make, model, year, trim, color, price_range, mileage, features. Stored in Redis session object. Merged across turns — enables multi-turn refinement ("I want something under $30k" → "also blue" → builds compound filter).
Track A — RAG Pipeline (FAQ / General)
Embed utterance with text-embedding-3-small → cosine similarity search over pgvector index of dealership knowledge base (FAQ docs, policies, service info, financing terms). Top-k=5 chunks retrieved, reranked with cross-encoder, injected into LLM context. Handles: "What are your hours?", "Do you offer financing?", "What's your return policy?"
Track B — Voice-to-SQL Agent (Vehicle Queries)
Extracted entities + schema description → LLM generates parameterized PostgreSQL query against the vehicles table. Query validated, sanitized (no raw interpolation), and executed. Results formatted as natural language: "We have 3 matching vehicles — a 2023 Toyota Camry XSE in Midnight Black at $28,995…". See Section 7 for full schema and query examples.
Response Synthesis
GPT-4o merges RAG chunks + SQL results into a conversational voice response. System prompt enforces: voice-first language (no markdown, no lists), max 3 vehicles per response, natural transitions, always end with a follow-up question to keep conversation going. Response trimmed to ~50 words for phone UX.
Human Transfer Gate
If intent is human_transfer, or confidence is low after 2 retries, or caller explicitly asks for a person: trigger Vonage connect action to transfer to live sales rep. Log conversation summary + entities to PostgreSQL for rep context before transfer.
06 Database Architecture — PostgreSQL + pgvector
// Primary Store
PostgreSQL Schema Design
Use PostgreSQL 16 with the pgvector extension. Hosted on Supabase or AWS RDS (Multi-AZ, 2-read replicas for query load). 
-- Core vehicle inventory table
CREATE TABLE vehicles (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  vin          VARCHAR(17) UNIQUE NOT NULL,
  make         VARCHAR(50)  NOT NULL,
  model        VARCHAR(100) NOT NULL,
  year         SMALLINT     NOT NULL,
  trim         VARCHAR(100),
  color_ext    VARCHAR(50),
  color_int    VARCHAR(50),
  mileage      INT,
  price        NUMERIC(10,2) NOT NULL,
  condition    VARCHAR(10), -- new/used/cpo
  transmission VARCHAR(20),
  fuel_type    VARCHAR(20),
  body_style   VARCHAR(30),
  features     JSONB,       -- ["AWD","Sunroof",...]
  status       VARCHAR(20), -- available/sold/hold
  description  TEXT,
  embedding    vector(1536), -- pgvector
  created_at   TIMESTAMPTZ DEFAULT now(),
  updated_at   TIMESTAMPTZ DEFAULT now()
);

-- HNSW index for fast ANN search
CREATE INDEX vehicles_embedding_idx
  ON vehicles USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);

-- Composite index for SQL filtering
CREATE INDEX idx_vehicles_search
  ON vehicles (make, model, year, price, status)
  WHERE status = 'available';

🔍
Hybrid Search Strategy: For vehicle queries, combine exact SQL filtering (price range, year, make/model) with pgvector cosine similarity on the vehicle description embedding for fuzzy natural-language matching ("something sporty but practical" → semantic match). Use PostgreSQL's native ability to combine WHERE clauses with <-> operator in a single query.
07 Voice-to-SQL Agent — Query Generator
// LLM-Generated SQL with Safety Guardrails
Query Generation Architecture
// System prompt for SQL generation (sent to GPT-4o)
const SQL_SYSTEM_PROMPT = `
You are a PostgreSQL query generator for a car dealership inventory system.
Generate ONLY SELECT queries. Never INSERT, UPDATE, DELETE, or DROP.
Always include WHERE status = 'available'.
Always use parameterized queries ($1, $2...) — never interpolate values.
Return JSON: { sql: string, params: any[], explanation: string }

Schema:
  vehicles(id, vin, make, model, year, trim, color_ext, color_int,
           mileage, price, condition, transmission, fuel_type,
           body_style, features JSONB, status, description, embedding)

User intent entities: {{ entities_json }}
`;

// Query validation layer
const validateQuery = (sql) => {
  const forbidden = ['INSERT', 'UPDATE', 'DELETE', 'DROP', 'ALTER', '--'];
  if (forbidden.some(f => sql.toUpperCase().includes(f)))
    throw new Error('Unsafe query rejected');
  if (!sql.includes("status = 'available'"))
    throw new Error('Missing availability filter');
};

// Example generated query for: "blue Toyota SUV under $35k"
SELECT make, model, year, trim, color_ext, price, mileage
FROM vehicles
WHERE status = 'available'
  AND make ILIKE $1
  AND body_style ILIKE $2
  AND color_ext ILIKE $3
  AND price <= $4
ORDER BY price ASC
LIMIT 5;
-- params: ['Toyota', '%SUV%', '%blue%', 35000]
// Query Cache
Redis Query Result Cache
Cache query results in Redis with a composite key: vehicle_query:{hash_of_entities}

TTL: 5 minutes — inventory changes frequently but callers rarely ask identical queries within a short window. Invalidated on vehicles table update via PostgreSQL LISTEN/NOTIFY trigger.

Estimated cache hit rate: 20–30% (common searches like "used trucks under $30k").
// Fallback
Query Fallback Strategy
If generated SQL returns 0 results: automatically widen constraints (remove one filter at a time in priority order: color → trim → year range expansion → price +20%).

If SQL generation fails validation: fall back to pre-built parameterized query templates for the 10 most common search patterns.

Always return at least 1 result or gracefully explain what similar options exist.
08 RAG Pipeline — Knowledge Base Architecture
// Ingestion
Document Ingestion Pipeline
  • Source docs: FAQ PDFs, policy docs, promotions, financing terms, service schedules
  • Chunking: recursive character splitting, 512 tokens, 50-token overlap
  • Embedding: text-embedding-3-small (1536 dims, fast + cheap)
  • Store chunks + embeddings in knowledge_chunks table
  • Re-index nightly via cron job for updated docs
// Retrieval
Vector Similarity Search
  • Embed query → cosine similarity search via HNSW index
  • Top-k=5 candidates retrieved (≈5ms with HNSW)
  • Rerank with cross-encoder: cross-encoder/ms-marco-MiniLM-L-6-v2
  • Final top-3 chunks injected into LLM context
  • Metadata filter: source IN ('faq','policy','promo')
// Generation
Grounded Response Generation
  • System prompt: "Answer using ONLY the provided context. If not found, say you'll have someone call back."
  • Hallucination guard: if no relevant chunk found (score < 0.75), trigger human transfer
  • Response grounded to retrieved text — no fabricated prices or policies
  • Conversation history (last 6 turns) included for context continuity
09 Redis — Caching, Sessions & Pub/Sub
Key Pattern Data Stored TTL Purpose
session:{call_uuid} JSON: caller info, entities, conversation history, TTS state 2 hours Per-call working memory
tts_cache:{sha256(text)} Binary PCM audio blob 24 hours Skip TTS API for repeated phrases
vehicle_query:{hash} JSON: SQL result set (vehicle list) 5 minutes Avoid redundant DB queries
embed_cache:{sha256(text)} Float32 vector (1536 dims) 1 hour Cache embeddings for repeated queries
rate_limit:{caller_phone} Call count counter 1 hour sliding Prevent abuse / runaway calls
pubsub:inventory_updated Pub/Sub channel N/A Invalidate vehicle_query caches on inventory change
lock:tts:{call_uuid} Distributed lock (SETNX) 5 seconds Prevent concurrent TTS generation for same call
📦
Redis Deployment: Use Redis 7+ with Redis Stack (includes RediSearch, RedisJSON). Deploy on AWS ElastiCache (cluster mode, 2 shards, 1 replica each) or Upstash Redis for serverless deployments. Enable maxmemory-policy: allkeys-lru with 4GB memory allocation.

12 Security & Compliance
// Auth & API Security
Authentication Layers
  • Vonage webhook validation: verify X-Vonage-Signature HMAC on every webhook
  • Internal service-to-service: mTLS or shared JWT (short-lived, 15min expiry)
  • Dashboard auth: NextAuth.js with GitHub OAuth or email/password (RBAC: manager/agent roles) — credentials stored in PostgreSQL
  • All API keys in Railway Environment Variables UI — never hardcoded in source or committed to GitHub
  • WebSocket connections: validate session token on upgrade handshake
  • SQL injection prevention: parameterized queries only, ORM for non-generated queries
// Privacy & Compliance
Call Recording & Data Handling
  • Play TCPA/CCPA disclosure at call start before any AI interaction
  • Call recordings encrypted at rest — stored to Cloudflare R2 (S3-compatible, cheap egress) via presigned URLs; 90-day retention policy
  • PII in PostgreSQL: phone numbers hashed after session ends (keep last 4 digits for UX)
  • Transcripts purged after 30 days unless flagged as lead
  • Redis session data: no PII stored beyond call_uuid and entities
  • SOC 2 Type II compliance path: audit logging for all data access
