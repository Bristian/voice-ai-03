01 Inbound Call Setup — Caller → AI Connected

The opening 15–20 seconds: caller dials sales rep's DID, the rep doesn't pick up, Vonage executes the NCCO, and the bidirectional audio WebSocket is established. This is every call's on-ramp.

sequenceDiagram autonumber participant C as 📞 Caller participant V as Vonage Voice API participant E as echokit-server participant R as Redis participant A as agent-api C->>V: Dial sales rep DID V->>V: Ring rep 15–20s (no answer) V->>E: POST /webhooks/answer
{from, to, uuid, conversation_uuid} E->>E: Validate X-Vonage-Signature (HMAC) E->>R: SET session:{call_uuid}
TTL 2h E-->>V: NCCO [talk + connect(websocket)] V->>C: Play "Please hold..." V->>E: WS upgrade /ws/voice
audio/l16;rate=16000 E->>A: POST /v1/agent/turn
(empty transcript, opening) A-->>E: greeting_text + TCPA disclosure E->>E: Look up tts_cache:{sha256(greeting)} Note over E,R: Cache HIT — skip OpenAI TTS call E->>V: Stream PCM (greeting) V->>C: Play greeting audio Note over C,V: Caller hears greeting within ~200ms of connect
⚡
Why the greeting feels instant: the opening phrase is pre-generated and stored in tts_cache:* with 24h TTL. First-call latency is effectively the WebSocket handshake plus Redis GET — under 50ms.
02 Conversational Turn — Utterance to Spoken Response

The hot path. Every turn after the greeting follows this shape: PCM frames → VAD → streaming STT → intent + parallel RAG/SQL → GPT-4o synthesis → streaming TTS. Target end-to-end: ≤ 800ms perceived.

sequenceDiagram autonumber participant C as Caller participant V as Vonage participant E as echokit-server participant EL as ElevenLabs STT participant A as agent-api participant OAI as OpenAI
(GPT-4o + TTS) participant PG as Postgres
+ pgvector participant R as Redis C->>V: Speak utterance V->>E: PCM frames (20ms / 320B) loop Every 20ms frame E->>E: Silero VAD (5ms, in-proc) E->>EL: Stream PCM chunk end EL-->>E: transcript_partial... Note over E: 600ms silence = utterance end EL-->>E: transcript_final (confidence ≥ 0.75) E->>A: POST /v1/agent/turn
{transcript, session_entities, history} par Track A — RAG (parallel) A->>R: GET embed_cache:{sha256(q)} alt cache miss A->>OAI: embeddings (text-embedding-3-small) OAI-->>A: 1536-dim vector A->>R: SET embed_cache end A->>PG: pgvector HNSW search (top-k=5) PG-->>A: 5 chunks A->>A: Cross-encoder rerank → top-3 and Track B — Voice-to-SQL A->>OAI: GPT-4o-mini
(intent + entities + SQL) OAI-->>A: {sql, params} A->>A: Validate SQL (guardrails) A->>R: GET vehicle_query:{hash} alt cache miss A->>PG: Execute parameterized query PG-->>A: vehicle rows A->>R: SET vehicle_query (TTL 5m) end end A->>OAI: GPT-4o synthesis
(RAG chunks + vehicles + history) OAI-->>A: response_text (≤ 50 words) A-->>E: {response_text, intent, vehicles, entities} E->>R: GET tts_cache:{sha256(response)} alt cache hit R-->>E: PCM blob else cache miss E->>OAI: TTS stream (tts-1, pcm 24kHz) OAI-->>E: PCM chunks (streamed) E->>E: ffmpeg → 16kHz mono E->>R: SET tts_cache (TTL 24h) end E->>V: Stream PCM out (first chunk ~150ms) V->>C: Play response audio
⚡
The par block is the whole ballgame. RAG and SQL must run concurrently via Promise.all / asyncio.gather. Running them serially blows the latency budget by ~200ms on complex queries. The synthesis LLM call is the only mandatory serial step afterward.
03 Barge-In — Caller Interrupts TTS Playback

If the caller starts speaking while the agent is mid-response, we must cancel TTS immediately and process the new utterance. Without this, the AI feels robotic. Silero VAD running locally makes the detection free (~5ms).

sequenceDiagram autonumber participant C as Caller participant V as Vonage participant E as echokit-server participant OAI as OpenAI TTS participant EL as ElevenLabs STT Note over E,OAI: Agent response is mid-stream OAI-->>E: PCM chunk N E->>V: Forward PCM chunk N V->>C: Play audio C->>V: Caller starts speaking (overlapping audio) V->>E: Inbound PCM frames E->>E: Silero VAD: isSpeech = true
AND session.ttsPlaying = true rect rgba(232, 86, 42, 0.08) Note over E: BARGE-IN TRIGGERED E-->>OAI: Abort TTS stream (cancel) E->>V: Send BARGE_IN control event V->>V: Halt playback to caller E->>E: Flush outbound buffer
Clear session.ttsPlaying end E->>EL: Begin streaming new utterance Note over E,EL: Normal turn proceeds from here
(see Sequence 02)
🔊
Thresholds matter: minimum speech duration 200ms (filters out coughs and background noise); end-of-utterance silence 600ms. Tune per deployment environment — noisy call-center backgrounds may need 250ms / 700ms.
04 RAG Pipeline — FAQ / Policy Questions

Zoomed in on Track A: a caller asks "Do you offer financing?" or "What are your hours?". Embedding, HNSW search, reranker, hallucination guard.

sequenceDiagram autonumber participant E as echokit-server participant A as agent-api participant R as Redis participant OAI as OpenAI participant PG as Postgres
(knowledge_chunks) participant CE as Cross-Encoder
(in-proc) E->>A: POST /v1/rag/query
{query: "do you offer financing?"} A->>R: GET embed_cache:{sha256(q)} alt cache miss A->>OAI: text-embedding-3-small OAI-->>A: vector(1536) A->>R: SET embed_cache (TTL 1h) else cache hit R-->>A: vector(1536) end A->>PG: SELECT id, source, content,
embedding <=> $1 AS distance
FROM knowledge_chunks
ORDER BY distance LIMIT 5 PG-->>A: top-5 chunks (HNSW, ~10ms) A->>CE: Rerank (query, chunks) CE-->>A: reordered, scored alt best_score < 0.75 Note over A: Hallucination guard TRIPPED A-->>E: {grounded: false,
answer: "Let me transfer you to a person"} else grounded A->>OAI: GPT-4o synthesis
(system: "answer ONLY from context") OAI-->>A: grounded answer A-->>E: {grounded: true, answer, chunks} end
🧭
The 0.75 threshold is load-bearing. It's the difference between an agent that confidently makes up a financing rate and one that transfers the caller to a human. Don't raise it without A/B testing transfer rate and caller satisfaction.
05 Voice-to-SQL — "Blue Toyota SUV under $35k"

Track B in detail: entities extracted across turns, SQL generated by GPT-4o-mini, validated by a hard-coded guardrail, executed with parameterized values, and fallback-widened if the result set is empty.

sequenceDiagram autonumber participant E as echokit-server participant A as agent-api participant R as Redis participant OAI as GPT-4o-mini participant V as SQL Validator participant PG as Postgres
(vehicles) E->>A: POST /v1/sql/search
{entities: {make:"Toyota",
body_style:"SUV", color:"blue",
price_max:35000}} A->>R: GET vehicle_query:{hash(entities)} alt cache hit (TTL 5m) R-->>A: cached result set A-->>E: vehicles[] else cache miss A->>OAI: Generate SQL + params
(schema-aware system prompt) OAI-->>A: {sql: "SELECT ... WHERE
status='available' AND make ILIKE $1
AND body_style ILIKE $2 ...",
params: ['Toyota','%SUV%','%blue%',35000]} A->>V: validateQuery(sql) alt unsafe V-->>A: throw (INSERT/UPDATE/DELETE
or missing availability filter) Note over A: Fall back to template query
for top-10 search patterns else safe V-->>A: ✓ end A->>PG: EXECUTE sql, params alt rows returned PG-->>A: vehicle[] A->>R: SET vehicle_query (TTL 5m) A-->>E: vehicles[] else 0 rows Note over A: Widen constraints in order:
color → trim → year → +20% price loop up to 4 retries A->>PG: EXECUTE widened SQL PG-->>A: result end A-->>E: vehicles[] + widened_constraints[] end end
🛡️
Validator is a security surface. It rejects any string containing INSERT|UPDATE|DELETE|DROP|ALTER|-- and requires WHERE status = 'available'. Plus parameterized-only — the LLM is told to use $1, $2 and the validator refuses interpolated values. Defense in depth.
06 Human Transfer — Escalating to a Sales Rep

Triggered by explicit caller request, repeated low confidence, or RAG hallucination guard tripping. Conversation summary + entities are persisted before the transfer so the human rep has context.

07 Supervisor Listen-In — Dashboard + WebRTC

A dealership manager opens the dashboard, picks an active call, and listens live via WebRTC while seeing the transcript stream in real time. Optionally takes over.

sequenceDiagram autonumber participant M as Manager
(Browser) participant D as dashboard
(Next.js) participant E as echokit-server
(mediasoup + Socket.io) participant V as Vonage participant C as Caller M->>D: Open /supervisor D->>D: NextAuth session check (role: manager) D->>E: Socket.io connect /ws/supervisor
(session token on handshake) E-->>D: Active calls list M->>D: Click call #abc123 D->>E: emit subscribe_call {call_uuid} par Live events stream E-->>D: transcript_partial E-->>D: transcript_final E-->>D: intent_classified E-->>D: entities_extracted E-->>D: vehicle_results D->>D: Zustand store updates
React re-renders and WebRTC audio bridge D->>E: RTC offer (SDP) via signaling WS E->>E: mediasoup creates consumer
for call_uuid producer E-->>D: RTC answer (SDP) D->>E: ICE candidates E-->>D: ICE candidates Note over D,E: DTLS-SRTP established E->>D: Forked live audio (encrypted) D->>M: Play through browser end Note over C,M: Manager hears caller + AI live.
Caller does NOT hear manager. opt Takeover M->>D: Click "Transfer to Me" D->>E: emit transfer_to_me
{call_uuid, manager_phone} E->>V: Vonage API: transfer call V->>M: Ring manager's phone Note over E: AI exits the loop. end
🌐
Audio is forked, not teed from the caller side. echokit-server is already decoding both directions for VAD/STT; mediasoup publishes the same PCM to an SFU channel per call_uuid. Supervisor joins as a consumer. Listen-only by default — mic is muted until explicit takeover.
08 Nightly Ingestion — Knowledge Base Refresh

One-shot Railway cron at 02:00 picks up any changed docs (FAQ, policies, promos, financing terms), chunks, re-embeds, and upserts. Separate from the hot path — if it fails, live calls still work with yesterday's embeddings.

sequenceDiagram autonumber participant Cron as Railway Cron
(0 2 * * *) participant W as ingestion-worker participant FS as Source Docs
(R2 / Google Drive) participant A as agent-api participant OAI as OpenAI
text-embedding-3-small participant PG as Postgres
(knowledge_chunks) Cron->>W: Start python ingest.py W->>FS: List source docs
(FAQ, policy, promo, financing) FS-->>W: doc list + last_modified loop For each changed doc W->>FS: Fetch content FS-->>W: text W->>W: Recursive char split
(512 tokens, 50 overlap) end W->>W: Batch chunks (≤ 100/batch) loop For each batch W->>A: POST /v1/embeddings
{texts: [...], cache: false} A->>OAI: batch embed OAI-->>A: vectors[] A-->>W: embeddings[] W->>PG: UPSERT knowledge_chunks
(id, source, content,
embedding, metadata) end W->>PG: REINDEX (optional, weekly) PG-->>W: ✓ W->>Cron: Exit 0 Note over Cron,PG: Live RAG traffic uses new
embeddings on next query.
🌙
Why a cron and not streaming? Dealership knowledge docs change slowly (policies monthly, FAQ weekly, promos weekly). Batch embedding is cheaper and simpler than CDC. If a promo needs same-day propagation, manually trigger railway run --service ingestion-worker python ingest.py.
